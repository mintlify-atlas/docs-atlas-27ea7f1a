  <ResponseField name="userId" type="number">

  <ResponseField name="companyId" type="number">

  </ResponseField>

  </ResponseField>

  <ResponseField name="assigned" type="array">

  </ResponseField>

  <ResponseField name="removed" type="array">

  </ResponseField>
</api>

## Replace User Permission Overrides

<api method="PUT" endpoint="/user-assignments/permissions/:idUsuario/context" auth="config:users:deny-permissions">
  Sets explicit permission overrides for a user in a specific company and application context. Overrides allow you to grant additional permissions (allow) or explicitly deny permissions (deny) that would otherwise be granted through roles.

  <ParamField path="idUsuario" type="number" required>
    The ID of the user
  </ParamField>

  <ParamField body="appCode" type="string" required>
    The application code context. Must be lowercase alphanumeric with hyphens.
  </ParamField>

  <ParamField body="companyId" type="number" required>
    The company ID context. Minimum value: 1.
  </ParamField>

  <ParamField body="allow" type="array">
    Array of permission codes to explicitly grant. Each code must follow the format `module:action[:subaction]` (e.g., "payroll:approve:special"). These permissions are added to those granted by roles.
  </ParamField>

  <ParamField body="deny" type="array">
    Array of permission codes to explicitly deny. Each code must follow the format `module:action[:subaction]`. Denials take precedence over both role-based and allowed permissions.
  </ParamField>

  <ResponseField name="userId" type="number">
    User ID
  </ResponseField>

  <ResponseField name="companyId" type="number">
    Company ID context
  </ResponseField>

  <ResponseField name="appCode" type="string">
    Application code context
  </ResponseField>

  <ResponseField name="allowed" type="array">
    Array of permission codes that were explicitly allowed
  </ResponseField>

  <ResponseField name="denied" type="array">
    Array of permission codes that were explicitly denied
  </ResponseField>
</api>

## Permission Resolution Logic

When determining if a user has a specific permission, the system follows this resolution order:

1. **Explicit Denials**: If the permission is in the user's `deny` list for the context, access is denied regardless of other factors.

2. **Explicit Allows**: If the permission is in the user's `allow` list, access is granted.

3. **Role-Based Permissions**: If the user has a role in the context that includes the permission, access is granted.

4. **Default**: If none of the above conditions match, access is denied.

This means denial overrides always take precedence, followed by explicit allows, and finally role-based permissions.

## Context-Based Access Control

The RBAC model is context-aware, meaning permissions are evaluated within the scope of:

- **User**: The person performing the action
- **Company**: The company being accessed
- **Application**: The application being used (kpital or timewise)

This allows for scenarios like:

- A user can be an ADMIN in Company A but an EMPLOYEE in Company B
- A user can have different roles in KPITAL vs TIMEWISE for the same company
- A user can have multiple roles within the same context
