---
title: "Verify Payroll"
api: "PATCH /payroll/{id}/verify"
description: "Verify a processed payroll, confirming snapshots and calculations are complete. Transitions from EN_PROCESO to VERIFICADA state."
---

## Authorization

Requires `payroll:verify` permission.

## Path Parameters

<ParamField path="id" type="number" required>
  Unique identifier of the payroll to verify.
</ParamField>

## Workflow

The verify operation performs validation checks before transitioning state:

1. **Validates state** - Only EN_PROCESO (Processing) payrolls can be verified
2. **Checks employee snapshots** - Ensures at least one employee snapshot exists
3. **Checks input snapshots** - Ensures at least one input snapshot exists
4. **Checks results** - Ensures at least one payroll result exists
5. **Updates state** - Transitions payroll to VERIFICADA (Verified)
6. **Increments version** - Updates `versionLock` for optimistic concurrency control

## State Transition

<ResponseField name="Before" type="string">
  `EN_PROCESO` (estado = 2)
</ResponseField>

<ResponseField name="After" type="string">
  `VERIFICADA` (estado = 3)
</ResponseField>

## Response

Returns the updated payroll calendar object.

<ResponseField name="id" type="number" required>
  The payroll identifier.
</ResponseField>

<ResponseField name="estado" type="number" required>
  Will be `3` (VERIFICADA) after successful verification.
</ResponseField>

<ResponseField name="versionLock" type="number" required>
  Version number incremented by 1.
</ResponseField>

<ResponseField name="fechaModificacion" type="string" required>
  Updated to current timestamp.
</ResponseField>

<ResponseField name="modificadoPor" type="number">
  User ID who verified the payroll.
</ResponseField>

All other payroll calendar fields are included in the response.

## Validation Requirements

For a payroll to be verified, it must have:

- At least one employee snapshot
- At least one input snapshot (salary, deductions, bonuses, etc.)
- At least one payroll result (calculated amounts)

These are created during the [process operation](/api/payroll/process).

## Next Steps

After verification, the payroll can:

- Be **applied** using the [apply endpoint](/api/payroll/apply) to make it immutable and update employee records
- Be **reopened** back to ABIERTA state if corrections are needed (requires `payroll:edit` permission)

## Notes

- Verification is a checkpoint ensuring the payroll is ready for application
- The VERIFICADA state indicates the payroll has been reviewed and approved
- A verified payroll can still be reopened if changes are needed before application
- Once applied, the payroll becomes immutable and cannot be reopened
- A domain event is emitted after successful verification for audit purposes
- The `versionLock` field is used to prevent concurrent modifications
